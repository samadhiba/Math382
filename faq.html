Frequently Asked Questions

❓ Why do you hash URLs instead of storing them directly?

We prioritize user privacy. Instead of storing raw URLs, we use SHA-256 hashes to anonymize the data. This ensures that your browsing or input remains secure while still enabling phishing detection through hash comparisons.

❓ How reliable is SHA-256 for phishing detection?

SHA-256 is a cryptographic hash function that provides a fixed-length output from any input. It’s collision-resistant, meaning it's unlikely two URLs will hash to the same value. While SHA-256 doesn’t *detect* phishing directly, it allows us to maintain a secure blacklist of known threats without storing sensitive URLs.

❓ How does CryptoPhish verify a threat?

We combine community-submitted reports with technical indicators like SSL certificate checks, domain age, and pattern matching. Our system flags known bad hashes and encourages human judgment through transparency, not black-box models.

❓ What happens when a URL is flagged?

If a submitted URL hash matches our database, it will display a red warning along with an explanation. If a user submits a new report, our team (or algorithm, depending on deployment stage) reviews it for indicators of phishing before adding it to the database.

❓ Can a “clean” result still be dangerous?

Yes. A “clean” status simply means the URL is not in our current hash database. Phishing is an evolving threat, so users should always proceed with caution. Look for signs of urgency, spelling mistakes, suspicious attachments, or unexpected requests for sensitive data.

❓ How does this project connect to cryptography concepts?

CryptoPhish applies concepts like entropy, collision resistance, and hash table design to ensure secure threat detection. We also plan to add math explainers on topics like modular arithmetic and SSL handshake validation for educational purposes.
